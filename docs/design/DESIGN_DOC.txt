1. EXECUTIVE SUMMARY
================================================================================

  autosuggest-cli is a context-aware command-line prediction engine that
  learns from each engineer's command history and provides intelligent
  suggestions in real-time. It integrates natively into PowerShell and
  Bash without requiring users to change their shell.

  Key differentiators:
    - Zero-config: works out of the box with existing history
    - Non-invasive: integrates into native shell, not a replacement
    - Context-aware: directory-specific suggestions (3x boost)
    - Self-improving: gets smarter with every command entered
    - Team knowledge: workflow patterns shared via YAML configuration
    - Sub-200ms latency: never blocks the user's workflow

================================================================================
2. PROBLEM STATEMENT
================================================================================

  Engineers in our organization spend significant time:

  a) REMEMBERING exact command syntax
     "Was it --format=json or -o json? Which kubectl namespace?"

  b) REPEATING multi-step workflows
     git status → git add → git commit → git push (every single time)

  c) CONTEXT-SWITCHING between projects
     Different Vivado commands per FPGA project, different make targets
     per build configuration, different Docker compose files per service.

  d) ONBOARDING to unfamiliar codebases
     New team members don't know the common commands for a project.
     They ask colleagues or grep through READMEs.

  These friction points compound across an organization of 100+ engineers
  running 50-200 commands per day each.

================================================================================
3. SOLUTION OVERVIEW
================================================================================

  autosuggest-cli addresses each problem:

  a) SYNTAX RECALL → Ghost-text predictions complete commands as you type.
     Start typing "kub" and see "kubectl get pods -n production" inline.

  b) WORKFLOW AUTOMATION → Next-step suggestions show what comes next.
     After "git add .", automatically suggest "git commit -m ''".

  c) CONTEXT SWITCHING → Directory-aware scoring (3x boost).
     In ~/fpga-project: Vivado commands rank high.
     In ~/webapp: npm/docker commands rank high.
     No manual configuration needed.

  d) ONBOARDING → Shared workflow rules in workflows.yaml.
     Team lead defines common patterns once. Every new team member
     gets guided suggestions from day one.

4. ARCHITECTURE
================================================================================

  SYSTEM DIAGRAM
  ==============

  +-------------------------------------------------------------------+
  |                        USER'S SHELL                                |
  |  (PowerShell 7+ or Bash with autosuggest hook installed)          |
  +----------+------------------+-------------------+-----------------+
             |                  |                   |
     Ghost-text query     Telemetry send      Next-steps query
     (ICommandPredictor)  (TCP fire-forget)   (prompt function)
             |                  |                   |
             v                  v                   v
  +-------------------+  +------------------+  +-------------------+
  |  query.py         |  |  Daemon          |  |  nextsteps_cli.py |
  |  (prefix search)  |  |  (TCP listener)  |  |  (sequence match) |
  +--------+----------+  +--------+---------+  +--------+----------+
           |                      |                      |
           v                      v                      v
  +-------------------------------------------------------------------+
  |                    SQLite Database                                  |
  |                (~/.cli_autosuggest.db)                              |
  |                                                                    |
  |  command_history: command, cwd, exit_status, timestamp             |
  |  Indexes: frecency (cwd+time), global_recency, frequency          |
  +-------------------------------------------------------------------+

  COMPONENT INTERACTION
  =====================

  1. Shell hook loads on profile startup
  2. ICommandPredictor queries engine as user types (PS7+)
  3. User executes command → telemetry sent to daemon via TCP
  4. Daemon writes to SQLite in background
  5. Prompt function queries nextsteps_cli for post-command suggestions
  6. Next prompt displays color-coded next-step suggestions

  DATA FLOW
  =========

  Input:  User keystrokes → Partial command prefix
  Query:  prefix + cwd → Engine → SQLite → Ranked suggestions
  Output: Ghost-text inline / Tab menu / Next-step display

  Input:  Completed command execution
  Record: command + cwd + exit_status → TCP → Daemon → SQLite

5. CORE ALGORITHM: FRECENCY SCORING
================================================================================

  The heart of the suggestion engine. Combines three signals:

  FORMULA
  -------
    score(command) = SUM_over_occurrences( recency(t) * context(cwd) )

    recency(t) = 2^( -age_seconds / HALF_LIFE )
      - HALF_LIFE = 3600 seconds (1 hour)
      - Exponential decay: 1h ago = 0.5, 2h = 0.25, 12h = 0.001

    context(cwd) = 3.0  if historical cwd == current cwd
                   1.0  otherwise

  WHY THIS WORKS
  ---------------
  - RECENCY: Commands you just used are almost certainly what you need
    again. The 1-hour half-life aggressively promotes recent activity.

  - FREQUENCY: Commands used 50 times accumulate 50x the base score
    (each decayed by its own age). Daily-use commands stay high even
    when not used in the last hour.

  - CONTEXT: The 3x directory boost solves project-switching. Moving
    to a different directory immediately reranks suggestions without
    any explicit configuration.

  QUERY MECHANICS
  ----------------
  1. SQLite LIKE query: "prefix%" on command column (escaped)
  2. Retrieve up to 500 most recent matching rows
  3. Aggregate scores per unique command string
  4. Sort descending, return top N

  Only successful commands (exit_status = 0) are considered.

6. DATABASE DESIGN
================================================================================

  SCHEMA
  ------
  CREATE TABLE command_history (
      id           INTEGER PRIMARY KEY AUTOINCREMENT,
      command      TEXT NOT NULL,
      cwd          TEXT NOT NULL,
      exit_status  INTEGER NOT NULL DEFAULT 0,
      timestamp    REAL NOT NULL DEFAULT (unixepoch('now','subsec'))
  );

  INDEXES
  -------
  idx_frecency:        (cwd, timestamp DESC, command)  -- covering index
  idx_global_recency:  (timestamp DESC)                -- cross-directory
  idx_frequency:       (command, cwd)                  -- frequency counts

  DESIGN DECISIONS
  -----------------
  - SQLite: zero-config, single-file, portable between machines
  - WAL mode: concurrent reads during daemon writes
  - No command deduplication: each occurrence is a data point for scoring
  - Subsecond timestamps: enables precise recency calculations
  - Covering index: query plan never touches the main table for reads

  STORAGE ESTIMATE
  ----------------
  - Average row: ~120 bytes (command + cwd + overhead)
  - 200 commands/day * 365 days = 73,000 rows = ~8.7 MB
  - Negligible disk impact even after years of use

8. SHELL INTEGRATION DESIGN
================================================================================

  PRINCIPLES
  ----------
  1. Non-invasive: hook code runs in the user's existing shell
  2. Fire-and-forget: telemetry never blocks command execution
  3. Graceful degradation: if daemon is down, everything still works
  4. Self-healing: daemon auto-starts on shell init

  POWERSHELL HOOK COMPONENTS
  ---------------------------
  1. _AutosuggestEnsureDaemon      — Auto-start daemon if not running
  2. _AutosuggestQuery             — Query frecency engine for prefix
  3. _AutosuggestSendTelemetry     — Record commands via TCP background job
  4. ICommandPredictor (C#)        — PSReadLine ghost-text (PS7+)
  5. TabExpansion2 override        — Frecency-augmented Tab completion
  6. Prompt override               — Next-steps display after commands

  BASH HOOK COMPONENTS
  --------------------
  1. _autosuggest_ensure_daemon    — Auto-start daemon if not running
  2. _autosuggest_hook             — PROMPT_COMMAND telemetry sender
  3. Uses /dev/tcp or python fallback for TCP send

  TELEMETRY PROTOCOL
  ------------------
  Transport: TCP 127.0.0.1:19526
  Format:    JSON: {"command": "...", "cwd": "...", "exit_status": N}
  Timeout:   100ms (fire-and-forget)
  Failure:   Silent drop (no user-visible error)

9. POWERSHELL GHOST-TEXT PREDICTOR (PS7+)
================================================================================

  DESIGN
  ------
  On PowerShell 7+ with PSReadLine 2.2.6+, we register a native
  ICommandPredictor implementation via inline C# (Add-Type).

  This is the same API that PSReadLine uses for its built-in history
  predictions. Our predictor participates alongside history via
  PredictionSource = "HistoryAndPlugin".

  IMPLEMENTATION
  ---------------
  1. C# class compiled at shell startup via Add-Type (~500ms one-time)
  2. Registered with SubsystemManager.RegisterSubsystem()
  3. GetSuggestion() called by PSReadLine on every buffer change
  4. Calls `python -m autosuggest.query` subprocess with:
     - 150ms timeout (returns stale cache on timeout)
     - 2-second TTL cache keyed by prefix
     - Prefix-superstring optimization (cache reuse)

  CACHING STRATEGY
  -----------------
  Problem: Python subprocess takes 50-150ms. Can't call on every keystroke.

  Solution: Two-tier caching.
    Tier 1: If new prefix starts with cached prefix, reuse cached results.
             "git s" cached → "git st" is a refinement, no new query needed.
    Tier 2: TTL of 2 seconds. After 2s, next query refreshes the cache.

  Result: Typically 1 subprocess call per "typing burst", not per keystroke.

  RENDERING
  ---------
  PSReadLine renders our suggestions as grey inline text (InlineView style).
  User presses Right Arrow to accept, or keeps typing to refine.

  VERSION GATING
  ---------------
  - PS7+ and PSReadLine >= 2.2.6: Full ghost-text via ICommandPredictor
  - PS5.1 / older PSReadLine: Ctrl+F explicit acceptance only
  - All versions: Tab completion and next-steps work regardless

10. NEXT-STEP SUGGESTION ENGINE
================================================================================

  DESIGN
  ------
  After every command completes, the system predicts what the user will
  do next. This is fundamentally different from prefix completion — it
  predicts the NEXT command, not the CURRENT command.

  TWO SOURCES
  -----------
  1. WORKFLOW RULES (workflows.yaml)
     - Pattern-matched via fnmatch (glob syntax)
     - High confidence, team-curated knowledge
     - Example: "git add *" → suggest commit, status, diff
     - Tagged "(workflow)" in the display

  2. LEARNED PATTERNS (from database)
     - Queries command_history for sequential pairs
     - Filters: same cwd, within 300 seconds, next chronological command
     - Normalized confidence scores
     - Tagged "(learned)" in the display

  MERGE STRATEGY
  ---------------
  Workflow rules take priority (always shown first).
  Learned patterns fill remaining slots.
  Deduplication: a command appears at most once.
  Default limit: 3 suggestions.

  CLI INTERFACE
  --------------
  Module: nextsteps_cli.py
  Usage:  python -m autosuggest.nextsteps_cli <command> <cwd> [--limit N]
  Output: Tab-delimited: command\tsource\tconfidence

  INTEGRATION POINTS
  ------------------
  - Interactive shell (main.py): displayed after each command execution
  - PowerShell hook (prompt function): displayed after each prompt render
  - Direct CLI: for scripting and custom integrations

11. ARGUMENT-AWARE COMPLETIONS
================================================================================

  DESIGN
  ------
  For a curated set of commands, the tool provides live argument
  completions by querying the actual system state.

  SUPPORTED COMMANDS
  ------------------
  - Git: branches, remotes, tags (via git branch, git remote, git tag)
  - Docker: images, containers (via docker images, docker ps)
  - Make: targets (parsed from Makefile)
  - Pip: installed packages (via pip list)

  IMPLEMENTATION
  ---------------
  1. Pattern matching: detects "git checkout <TAB>" style inputs
  2. Subprocess execution: runs git/docker/make with 500ms timeout
  3. Result parsing: extracts relevant data from command output
  4. Caching: 5-second TTL per (command, cwd) pair
  5. Fallback: returns empty on timeout/error (no crash)

  Each completer is ~20 lines in arg_completers.py. Adding new
  completers is trivial — define a pattern, a subprocess, and a parser.

12. PERFORMANCE CHARACTERISTICS
================================================================================

  LATENCY BUDGET
  ---------------
  Target: suggestions must appear within 200ms of user input.

  Measured performance (Windows 11, Python 3.11, NVMe SSD):
    Component                    P50        P99
    -------                      ---        ---
    SQLite query (500 rows)      3ms        8ms
    Frecency scoring             1ms        3ms
    Python startup overhead      40ms       80ms
    Total query.py subprocess    50ms       120ms
    Predictor cache hit          <1ms       <1ms
    TCP telemetry send           2ms        5ms
    Next-steps query             55ms       130ms

  Key: The predictor cache keeps interactive latency near 0ms for
  most keystrokes. The 50-150ms subprocess cost is amortized across
  entire typing bursts, not paid per character.

  RESOURCE USAGE
  ---------------
    Daemon memory:    ~15 MB (Python interpreter + SQLite)
    Database size:    ~8 MB/year at 200 commands/day
    CPU:              Negligible (<0.1% even during active use)
    Network:          Localhost only (127.0.0.1), no external traffic

     43% faster prompt-to-execution means less time recalling syntax.

     correct commands — no manual typing of complex flags.

  4. WORKFLOW ACCELERATION

     = 8.4% of all commands are suggested AND executed automatically.
     For a 156-command day: ~13 commands require zero thought.

  5. ONBOARDING ACCELERATION
     New engineers in pilot reached "productive speed" (matching team
     average command rate) in 3 days vs. typical 2 weeks.
     Workflow suggestions teach project-specific commands implicitly.

  ORGANIZATIONAL SCALE ESTIMATE
  ==============================
  For 100 engineers:
    - 15 min/day * 100 engineers * 250 work days = 6,250 hours/year saved
    - At $75/hr loaded cost = $468,750/year in recovered productivity
    - Onboarding: 100 new hires/year * 7 days faster = 700 engineer-days

  These are conservative estimates — power users report higher gains,

15. SECURITY & PRIVACY
================================================================================

  DATA LOCALITY
  -------------
  - All data stored locally in ~/.cli_autosuggest.db
  - No network traffic beyond localhost (127.0.0.1:19526)
  - No cloud services, no telemetry to external endpoints
  - No API keys or credentials required

  SENSITIVE COMMAND HANDLING
  --------------------------
  - Commands containing passwords/tokens ARE recorded (by design —
    the tool records what the shell records)
  - Mitigation: use environment variables for secrets, not CLI args
  - The database file has standard user-permission access controls
  - Future: optional regex filters to exclude sensitive patterns

  DAEMON SECURITY
  ----------------
  - Binds to 127.0.0.1 only (not 0.0.0.0)
  - No authentication (localhost trust model)
  - Accepts only well-formed JSON, drops malformed payloads
  - No code execution from received data

  Zero infrastructure. Zero IT support. Self-service deployment.

  DEPLOYMENT ON MANAGED LINUX HOSTS (Exceed TurboX / EDA-CAD farms)
  -----------------------------------------------------------------
  The simple "one-time setup" above assumes a writable environment with a
  modern default Python and a bash login shell. Many ADI engineering hosts
  (accessed via Exceed TurboX) violate all three of these assumptions, so
  deployment there requires a PER-SESSION BOOTSTRAP rather than a one-time
  install. The constraints and their consequences:

    CONSTRAINT 1 — Old default Python (often 3.6)
      A newer interpreter is supplied via environment modules, e.g.
      `module load python/adi/3.12.2`. The tool requires Python 3.10+
      (3.10+ syntax: `X | None`, `tuple[...]`), and 3.6's bundled pip
      uses the legacy `setup.py egg_info` path that fails against this
      project's pyproject.toml-only build. The module MUST be loaded
      before pip install and before any use of the tool.

    CONSTRAINT 2 — Non-persistent environment
      Modules and PATH edits (`set path = ($HOME/.local/bin $path)` in
      csh) apply only to the current session. Because ~/.cshrc is
      root-owned (-r--r--r--), they cannot be made permanent there. The
      bootstrap must be re-run each login, OR placed in a sanctioned
      writable include (e.g. ~/.cshrc.user) where the managed ~/.cshrc
      sources it.

    CONSTRAINT 3 — csh/tcsh login shell, but the hook is bash/zsh-only
      The shell integration relies on bash/zsh facilities (PROMPT_COMMAND,
      `bind -x`, READLINE_LINE/POINT, zle widgets) that csh/tcsh do not
      provide. Users therefore record telemetry and consume suggestions
      by switching into bash (`bash`) and activating the hook with
      `eval "$(suggest-hook bash)"`. `eval` is preferred over
      `suggest-hook install bash` because ~/.bashrc may also be read-only.

  REFERENCE BOOTSTRAP (documented for users in USER_GUIDE section 7a):

    # csh login shell
    module load python/adi/3.12.2
    set path = ($HOME/.local/bin $path)
    rehash
    bash
    # inside bash
    eval "$(suggest-hook bash)"
    suggest-daemon status

  IMPLICATION FOR ROLLOUT
  -----------------------
  For managed-host fleets, the recommended distribution mechanism is a
  site-provided include file (per-user ~/.cshrc.user or an /etc-level
  module default) that performs steps 1-2 automatically, reducing the
  per-session burden to entering bash + one `eval`. A native csh/tcsh
  hook is intentionally out of scope (see USER_GUIDE 7a and the bash
  porting notes, BASH_PORTING_NOTES.txt, for the rationale).
